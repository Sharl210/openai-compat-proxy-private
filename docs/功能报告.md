# openai-compat-proxy 功能报告

## 项目目标

本项目是一个 Go 单二进制 OpenAI 兼容代理，用来把一个行为不稳定、兼容性不完整的上游 Responses 接口包装成更稳定的下游接口。

当前主推接口：

- `POST /v1/chat/completions`

当前仍保留的兼容/辅助接口：

- `GET /v1/models`
- `GET /healthz`

## 已实现能力

### 1. OpenAI 兼容入口

- 当前主要维护 `chat/completions` 单端口体验
- 支持 `models` 透传入口
- 所有主线请求内部统一转发到上游 `/v1/responses`
- 下游 `stream=false` 时由代理自行聚合上游 SSE 后返回完整 JSON

### 2. 鉴权能力

- 支持代理自身访问 key：`Authorization: Bearer <proxy-key>`
- 支持调用方显式传上游 key：`X-Upstream-Authorization: Bearer <upstream-key>`
- 支持服务端默认上游 key：`UPSTREAM_API_KEY`

### 3. 请求映射能力

- Chat 文本消息 → Responses `input` list
- Responses `input` → 统一 canonical model
- 支持 text input
- 支持 image input / `image_url`
- 支持 tools / function calling 定义透传
- 支持 reasoning 字段透传
- chat 请求中的 reasoning 设置按客户端意图透传，不做强制注入

### 4. 响应聚合能力

- 聚合 `response.output_text.delta`
- 聚合 `response.function_call_arguments.delta`
- 聚合 `response.output_item.done` 中的 function metadata
- 聚合 reasoning 相关事件
- 对 `/v1/chat/completions` 输出标准 `tool_calls`
- 对 `/v1/chat/completions` 输出扩展字段 `reasoning_content`
- 对 `/v1/chat/completions` 输出 `usage.completion_tokens_details.reasoning_tokens`
- 对 `/v1/chat/completions` 保留 `usage.prompt_tokens_details.cached_tokens`

### 5. 稳定性处理

- 上游 5xx 错误转为标准代理错误
- 对瞬时上游 5xx 做有界重试与退避
- 上游挂死时走超时保护，避免客户端一直卡住
- 未知路由返回统一 OpenAI 风格错误体
- 所有请求返回 `X-Request-Id`
- 规范化主线路由返回 `X-Proxy-Normalization-Version: v1`

### 6. 流式与边界增强

- `chat/completions` 路由的 `stream=true` 已支持真正边读边写的 `chat.completion.chunk`
- `chat` 流式工具调用已支持先发 tool metadata，再发 arguments 增量
- `chat` 在正文开始前支持轻量状态流，缓解长时间空白
- `responses` 非流式已支持保留多模态 output content parts
- `chat` 非流式遇到无法安全映射的多模态 output 时，会明确返回 `unsupported_output_mapping`

## 真实联调结论

### 已测通

基于当前真实上游 `https://api-vip.codex-for.me/v1` 和提供的测试 key，已经实测通过：

#### 1. 文本 chat

- 代理地址：`http://127.0.0.1:18082/v1/chat/completions`
- 模型：`gpt-5`
- 返回：`200`

示例结果：

```json
{
  "object": "chat.completion",
  "choices": [
    {
      "index": 0,
      "finish_reason": "stop",
      "message": {
        "role": "assistant",
        "content": "Hello"
      }
    }
  ]
}
```

#### 2.1 多轮 assistant 历史兼容

- 代理入口：`/v1/chat/completions`
- 模型：`gpt-5` / `gpt-5.4`
- 场景：调用方携带多轮历史，其中包含 assistant 文本消息

当前版本在转发到上游 `/v1/responses` 时，已经按角色重写文本 content type：

- `user` / `system` / `developer` 文本 → `input_text`
- `assistant` 文本 → `output_text`

这用于兼容部分上游对 assistant 历史消息的严格校验，避免出现：

- `Invalid value 'input_text'. Supported values are: 'output_text' and 'refusal'`
- `param input[n].content[0]`

#### 3. 多模态输入

- 代理入口：`/v1/chat/completions`
- 模型：`gpt-5`
- 输入：text + image_url
- 返回：`200`

已实测请求一张公开图片 URL，代理返回答案：`Python`。

#### 4. 工具调用

- 代理入口：`/v1/chat/completions`
- 模型：`gpt-5`
- 输入：function tool `get_weather`
- 返回：`200`
- 返回体中包含标准 `tool_calls`

已实测得到：

```json
{
  "tool_calls": [
    {
      "id": "call_fvwx4X7255I2rQuTFYiCLuqh",
      "type": "function",
      "function": {
        "name": "get_weather",
        "arguments": "{\"city\":\"Shanghai\"}"
      }
    }
  ]
}
```

#### 5. models 透传（兼容保留）

- 代理入口：`GET /v1/models`
- 行为：透传到上游 `/v1/models`
- 返回：`200`

当前代理不会重写模型列表，只做鉴权和透传。

#### 6. chat reasoning_content 扩展

- 代理入口：`/v1/chat/completions`
- 场景：上游返回 `response.reasoning.delta`

当前版本会按兼容生态常见扩展，把 reasoning 单独暴露为：

- non-stream: `choices[0].message.reasoning_content`
- stream: `choices[0].delta.reasoning_content`

它不会把 reasoning 混进普通 `content`，因此支持扩展字段的客户端可以显示思维内容，而不支持的客户端仍只会显示最终回答。

当前可见 reasoning 既支持来自 `response.reasoning.delta` 的明文增量，也支持来自 reasoning output item `summary[]` 中 `summary_text` 的汇总文本。

此外，为了改善首字前的空白期，代理还会在正文 token 出现前发出轻量状态提示，例如：

- `分析中…`
- `正在组织回答…`

这些提示同样通过 `delta.reasoning_content` 发出，但它们只会在**上游没有返回真实可见 reasoning**时启用；如果上游已经给了可见 reasoning，代理不会覆盖它。

#### 6.1 reasoning 请求透传策略

当前版本的代理策略是：

- 不替调用方强行打开推理
- 不在客户端未请求时私自追加 `reasoning` 配置
- 如果调用方只传 `reasoning_effort`，则兼容映射到上游 `reasoning.effort`
- chat 兼容请求默认补 `reasoning.summary: "auto"`
- 如果调用方直接传 `reasoning` 对象，则尽量原样透传到上游；若缺少 `summary`，代理会补 `summary: "auto"`

这意味着 chat 兼容层会代替客户端补足上游 `/responses` 所需的 summary 请求设置，但不会在客户端完全未请求推理时擅自开启推理。

#### 7. chat reasoning_tokens 兼容

- 代理入口：`/v1/chat/completions`
- 场景：上游只返回加密 reasoning，但在 completed usage 中返回 `reasoning_tokens`

当前版本会把它映射为：

- non-stream: `usage.completion_tokens_details.reasoning_tokens`
- stream: 最后一个 usage chunk 中的 `usage.completion_tokens_details.reasoning_tokens`

#### 7.1 chat cached_tokens 可观测性

- 代理入口：`/v1/chat/completions`
- 场景：上游 usage 返回 `input_tokens_details.cached_tokens`

当前版本会把它映射为：

- non-stream: `usage.prompt_tokens_details.cached_tokens`
- stream: 最后一个 usage chunk 中的 `usage.prompt_tokens_details.cached_tokens`

这样客户端就能直接观察“是否命中上游缓存”，而不只是看到普通 token 统计。

#### 8. normalization version 契约

为了保证“始终走代理”场景下的缓存稳定性，当前版本把规范化规则显式版本化。参与规范化的主线路由会返回：

- `X-Proxy-Normalization-Version: v1`

`v1` 代表当前固定的请求规范化合同，包括：

- 上游统一走 `/responses`
- 上游统一 `stream=true`
- `assistant` 文本历史映射为 `output_text`
- `reasoning_content` 仅作为下游展示扩展字段，不再回放到上游 assistant prompt
- chat reasoning 缺少 `summary` 时补 `summary: "auto"`
- tool schema 缺少 `items` 的 array 节点自动补 `items: {}`

如果未来需要调整这些缓存前缀敏感规则，应视为 normalization version 变更，而不是静默修改 `v1` 行为。

#### 8.1 chat 多轮增量缓存修复

- 代理入口：`/v1/chat/completions`
- 场景：真实多轮对话逐条追加时，客户端会看到 `reasoning_content`，但这个字段不应该污染上游缓存前缀

当前版本已修正：

- `reasoning_content` 保留给下游显示
- 但不会再被回放到上游 assistant `output_text`

这样可以减少“重复整包请求能命中，但真实多轮逐条追加时命中下降”的代理侧根因。

- non-stream: `usage.completion_tokens_details.reasoning_tokens`
- stream: 当请求带 `stream_options.include_usage: true` 时，在最终 usage chunk 中返回

这用于兼容“推理发生了但正文不可见”的上游实现。

#### 8. chat 真正流式转发

当前版本已经修正了 `chat/completions` 之前“先完整读取上游 SSE，再统一输出”的架构问题。

现在的行为是：

- 上游 event 到达后立即转译为下游 `chat.completion.chunk`
- 每个 chunk 立刻 flush
- 收到 completed 事件后再发送 usage chunk（如果请求带 `stream_options.include_usage=true`）和 `[DONE]`

这解决了“长时间没首字，然后一下子吐很多 token”的主要代理层根因。

#### 9. 默认脱敏日志系统

当前版本新增双通道日志：

- stdout 摘要日志
- 本地 JSON 文件日志

默认记录：

- 下游请求
- canonical 请求摘要
- 上游请求
- 上游 usage / `cached_tokens`
- 下游响应

默认脱敏：

- `Authorization`
- body 原文

新增环境变量：

- `LOG_FILE_PATH`
- `LOG_INCLUDE_BODIES`
- `LOG_MAX_SIZE_MB`
- `LOG_MAX_BACKUPS`

默认日志文件路径：

- `.proxy.requests.jsonl`

日志现在还会记录更适合排缓存问题的摘要字段，包括：

- `message_hashes`
- `prefix_hashes`
- `input_item_hashes`
- `input_prefix_hashes`

并且支持按文件大小自动轮转和归档清理。

#### 10. restart 脚本

新增：

- `scripts/restart-linux.sh`

行为保持最小：

1. 执行 `scripts/uninstall-linux.sh`
2. 执行 `scripts/deploy-linux.sh`

## 上游实际限制

这部分是通过真实探测得出的，不是猜测。

### 当前可用模型

`/v1/models` 返回的可用模型包括：

- `gpt-5`
- `gpt-5.4`
- `gpt-5.1`
- 其他若干 `gpt-5.*` / `codex-*`

### 当前已确认的有效组合

上游 `/v1/responses` 至少在下面组合下可用：

- `model = gpt-5`
- `model = gpt-5.4`
- `input` 必须是 **list**，不能直接是字符串
- `stream = true`
- 多轮 assistant 历史消息需要以 assistant output 语义重放，不能把 assistant 文本当成 `input_text`
- tools 参数里的 array schema 如果缺少 `items`，需要在转发前做兼容补全

### 当前不建议的组合

- `model = gpt-4.1`：不在上游返回的模型列表里
- `input` 直接传字符串：上游会报 `Input must be a list`

## 当前仍存在的边界

- `chat` 流式工具调用已经可用，但与部分 OpenAI 官方 SDK 的细节行为仍可能存在实现差异
- `chat` 的 `reasoning_content` 是兼容生态扩展字段，不是 OpenAI 官方 `chat/completions` 标准字段
- 如果上游既没有明文 reasoning delta，也没有 reasoning summary，只返回 `encrypted_content`，代理无法凭空生成可见 reasoning 正文
- 如果上游本身很久才返回第一个正文 token，首字延迟仍会存在；当前版本只修复代理层缓冲，不改变上游生成速度
- `chat` 路由对复杂多模态输出仍采取“安全失败”策略，而不是激进降级
- 如果调用方要把上游原生 Responses 历史原样回放进 `/v1/responses`，当前入站仍未完整接受 `output_text` / `refusal` 等 assistant output part
- 如果图片 URL 本身被上游拉取时返回 403，则代理会把错误透传出来，这不属于代理协议兼容问题

## 推荐使用方式

如果你要稳定使用当前代理，建议默认：

- `model` 用 `gpt-5` 或 `gpt-5.4`
- 图片输入走 `chat/completions`
- tool calling 也走 `chat/completions`
- 文本、流式、tool、reasoning 相关能力优先走 `chat/completions`
- `responses` 仅作为兼容保留入口，不再作为主线推荐入口
