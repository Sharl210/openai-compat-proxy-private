# openai-compat-proxy 部署文档

## 1. 部署形态

本项目是 Go 单二进制服务，不依赖数据库或外部状态存储。

当前推荐部署形态：

- 直接运行二进制
- 或挂到 `systemd`
- 或再放到 Nginx / Caddy 后面做反代

## 2. 构建

项目目录：

```bash
/home/harl/projects/openai-compat-proxy
```

构建命令：

```bash
go build -o bin/openai-compat-proxy ./cmd/proxy
```

构建产物：

```bash
bin/openai-compat-proxy
```

## 3. 环境变量

最小运行所需环境变量：

```bash
LISTEN_ADDR=:18082
UPSTREAM_BASE_URL=https://api-vip.codex-for.me/v1
UPSTREAM_API_KEY=<your-upstream-key>
```

可选环境变量：

```bash
PROXY_API_KEY=<your-proxy-key>
```

## 4. `.env.example` 与正式 `.env`

项目现在提供：

```bash
.env.example
```

部署脚本不会直接读取 `.env.example`，而是**必须要求存在正式 `.env`**。

准备方式：

```bash
cp .env.example .env
```

然后至少填写：

```bash
LISTEN_ADDR=:18082
UPSTREAM_BASE_URL=https://api-vip.codex-for.me/v1
UPSTREAM_API_KEY=<your-upstream-key>
```

## 5. 一键 Linux 部署脚本

项目现在提供：

```bash
scripts/deploy-linux.sh
scripts/uninstall-linux.sh
```

使用方法：

```bash
chmod +x scripts/deploy-linux.sh
./scripts/deploy-linux.sh
```

行为：

- 检查 `.env` 是否存在，不存在则直接失败
- 读取 `.env` 中的正式配置
- 构建最新二进制
- 停掉旧进程（如果有）
- 后台启动最新服务
- 自动做一次 `healthz` 检查

如果脚本成功，会输出：

- 二进制路径
- 新进程 PID
- health 检查地址

### 卸载运行中的服务

如果你只想停止服务并清理运行时产物，而**不删除仓库目录**，执行：

```bash
chmod +x scripts/uninstall-linux.sh
./scripts/uninstall-linux.sh
```

该脚本会：

- 停止 `.proxy.pid` 中记录的进程
- 删除 `.proxy.pid`
- 删除 `.proxy.log`
- 删除 `bin/openai-compat-proxy`

它不会删除：

- 源码目录
- `.env`
- 文档
- tests

### 后台启动

nohup env \
  LISTEN_ADDR=:18082 \
  UPSTREAM_BASE_URL=https://api-vip.codex-for.me/v1 \
  UPSTREAM_API_KEY=<your-upstream-key> \

## 7. systemd 部署示例

创建服务文件：

```ini
[Unit]
Description=openai-compat-proxy
After=network.target

[Service]
WorkingDirectory=/opt/openai-compat-proxy
ExecStart=/opt/openai-compat-proxy/bin/openai-compat-proxy
Restart=always
RestartSec=3
Environment=LISTEN_ADDR=:18082
Environment=UPSTREAM_BASE_URL=https://api-vip.codex-for.me/v1
Environment=UPSTREAM_API_KEY=<your-upstream-key>
# Environment=PROXY_API_KEY=<your-proxy-key>

[Install]
WantedBy=multi-user.target
```

生效命令：

```bash
sudo systemctl daemon-reload
sudo systemctl enable openai-compat-proxy
sudo systemctl restart openai-compat-proxy
sudo systemctl status openai-compat-proxy
```

## 8. 接口验证

### health

```bash
curl http://127.0.0.1:18082/healthz
```

预期：

```json
{"status":"ok"}
```

### responses

```bash
curl http://127.0.0.1:18082/v1/responses \
  -H 'Content-Type: application/json' \
  -d '{
    "model": "gpt-5",
    "input": [{
      "role": "user",
      "content": [{"type": "input_text", "text": "Say hello in one word."}]
    }],
    "stream": false
  }'
```

### chat

```bash
curl http://127.0.0.1:18082/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{
    "model": "gpt-5",
    "messages": [
      {"role": "user", "content": "Say hello in one word."}
    ],
    "stream": false
  }'
```

### chat 多轮历史

如果调用方会把之前的 assistant 回复一并带回 `/v1/chat/completions`，当前版本会在代理转发到上游 `/v1/responses` 时按角色重写文本 part：

- `user` / `system` / `developer` 文本 → `input_text`
- `assistant` 文本 → `output_text`

这样可以兼容会对 assistant 历史消息做严格校验的上游实现。

### chat reasoning_content 扩展

当前版本对 `/v1/chat/completions` 增加一个兼容生态常见的扩展字段：

- non-stream: `choices[0].message.reasoning_content`
- stream: `choices[0].delta.reasoning_content`

它不会污染标准 `content` 文本，而是把推理内容单独暴露给支持该扩展的客户端/UI。

如果你的客户端本身不会读取这个扩展字段，即使上游有 reasoning 事件，也可能仍然只显示最终回答。

当前可见 reasoning 的提取来源包括：

- `response.reasoning.delta`
- reasoning output item 的 `summary[] -> summary_text`

如果上游只有加密 reasoning 而没有可见 summary，代理不会伪造 `reasoning_content`。

### chat reasoning_tokens

如果上游在完成事件里返回 usage 统计，当前版本会把推理 token 数映射为：

- non-stream: `usage.completion_tokens_details.reasoning_tokens`
- stream: 当请求带 `stream_options.include_usage: true` 时，在最后一个 usage chunk 中返回

这样即使 reasoning 正文不可见，客户端也能知道这次请求是否消耗了 reasoning tokens。

### chat stream_options.include_usage

当前版本支持：

```json
{"stream_options":{"include_usage":true}}
```

开启后，流式 `chat/completions` 会在 `data: [DONE]` 之前额外发送一个 `choices: []` 的 usage chunk。

### models

```bash
curl http://127.0.0.1:18082/v1/models \
  -H 'Authorization: Bearer <proxy-key>'
```

当前实现会把 `GET /v1/models` 请求透传到上游 `/v1/models`，并回传上游 JSON 结果。

### tools schema 兼容

当前版本会在把 tools/function 定义发给上游前，做一个最小 JSON Schema 兼容修复：

- 当某个 schema 节点声明 `"type": "array"`
- 但缺少 `items`
- 代理会自动补成 `"items": {}`

这可以规避部分上游返回的 `invalid_function_parameters` / `array schema is missing items` 错误。

### 多模态

```bash
curl http://127.0.0.1:18082/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{
    "model": "gpt-5",
    "messages": [{
      "role": "user",
      "content": [
        {"type": "text", "text": "What language logo is this? Answer in one word."},
        {"type": "image_url", "image_url": {"url": "https://raw.githubusercontent.com/github/explore/main/topics/python/python.png"}}
      ]
    }],
    "stream": false
  }'
```

### 工具调用

```bash
curl http://127.0.0.1:18082/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{
    "model": "gpt-5",

```

## 9. 生产建议

- 默认使用 `gpt-5` 或 `gpt-5.4`
- 不建议继续把 `gpt-4.1` 当默认模型
- `responses` 上游请求体应保持 `input` 为 list
- 如果要对外开放，建议设置 `PROXY_API_KEY`
- 如果要公网提供服务，建议再接一层 Nginx / Caddy 做 TLS 和限流

## 10. 故障排查

### 上游报 `Input must be a list`

说明调用方发给 `responses` 的形状不符合当前上游要求。

### 上游报图片下载 403

说明上游无法访问该图片 URL，不是代理入口本身错误。

### 上游报 `no clients available`

说明上游后端资源暂时不可用；当前代理已经实现有界重试，但如果连续失败，仍会把错误规范化返回。

### 上游报 `Invalid value 'input_text'`

如果报错位置类似 `input[n].content[0]`，通常说明某条 assistant 历史消息被错误当成输入文本重放。

当前版本的 `/v1/chat/completions` 代理已经按角色修正这个映射；如果你仍遇到这个错误，优先检查：

- 是否仍在运行旧二进制
- 是否通过 `scripts/deploy-linux.sh` 重新部署了最新版本
- 是否直接调用了 `/v1/responses` 并自行传入了不兼容的 assistant content part

### 上游报 `array schema is missing items`

如果错误位置类似 `tools[n].parameters`，通常说明某个 tool/function 参数 schema 里声明了数组类型，但没有给 `items`。

当前版本会自动补一个最小兼容形状 `items: {}`；如果你仍遇到这个错误，优先检查：

- 是否仍在运行旧二进制
- 是否已经重新部署最新版本
- 是否上游还在校验其他更严格的 schema 约束

### 为什么有推理但看不到 reasoning_content

如果你确认模型用了推理，但 `chat` 里没有 `reasoning_content`，常见原因是：

- 上游只返回了 `encrypted_content`
- reasoning item 的 `summary` 为空
- 客户端没有读取 `message.reasoning_content` / `delta.reasoning_content`

这种情况下，仍然可以通过 `usage.completion_tokens_details.reasoning_tokens` 判断这次请求是否发生了推理。

### chat 真正流式转发

当前版本的 `chat/completions` 已改为**边收上游 event，边写下游 chunk**。

这意味着：

- 不再等待上游完整结束后才统一输出
- 一旦上游开始返回正文 token，下游会尽快收到对应 `delta.content`
- `stream_options.include_usage=true` 时，usage chunk 会在最后单独发送

如果你仍感觉“很久没首字，然后短时间内一下子很多内容”，优先检查：

- 上游本身是否很久才开始吐首个正文 token
- 前面是否只有 reasoning / 状态事件而没有正文 token
- 部署层是否还有额外反向代理缓冲

### chat 预正文状态流

为了减少正文前长时间空白，当前版本会在正文真正开始前额外发送轻量状态提示：

- `分析中…`
- `正在组织回答…`

这些提示通过 `delta.reasoning_content` 发出；一旦正文开始，就停止发送。

注意：这属于代理层体验优化，不是模型原始推理内容。

### chat 反缓冲响应头

当前 `chat` 流式响应会显式设置：

- `Cache-Control: no-cache, no-transform`
- `Connection: keep-alive`
- `X-Accel-Buffering: no`

如果你部署在 Nginx、Caddy、Cloudflare 或其他网关后面，仍建议继续检查这些层是否开启了响应缓冲。

### chat reasoning 请求透传

当前版本不会在代理层强行替调用方打开推理。

具体规则：

- 调用方**没传** `reasoning` / `reasoning_effort` → 代理不主动打开推理
- 调用方传了 `reasoning_effort` → 代理转换为上游 `reasoning.effort`
- chat 兼容请求默认补 `reasoning.summary: "auto"`
- 调用方传了完整 `reasoning` 对象 → 代理尽量按原样透传；若缺少 `summary`，代理补 `summary: "auto"`

如果你希望显式指定完整 reasoning 对象，仍然可以直接传入，例如：

```json
{"reasoning":{"effort":"high","summary":"auto"}}
```
